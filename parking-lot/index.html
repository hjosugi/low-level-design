マルチ入口のレースコンディション検証

実際の駐車場システムでは、複数のゲート（入口）から同時に車両が入庫を試みることがあります。 適切な排他制御（Javaでの ReentrantLock または synchronized）を行わないと、 「2台の車両が同時に1つの同じ駐車スポットを確保し、別々のチケットで割り当てられる」というレースコンディション重大障害が発生します。

💡 このシミュレーターで検証できること:
  • 空き枠が「残り1つ」の過酷な条件下で、3つのゲートから同時にCarが入庫したケース。
  • 排他ロックなし (No Lock): スレッドが「空き枠検索」と「確保(occupied)」を順序なく交互に実行するため、重複割り当てエラーが発生する様子を視覚化。
  • 排他ロックあり (ReentrantLock): 1つのゲートがロックを保持している間、他のゲートのスレッドは待機し、安全に拒否されるフロー。
排他制御 (ReentrantLock) : 有効 スレッド排他実行によりレースコンディションを防止します

Vehicle はクラスにしない

Design Decision

一番やりがちなミスは、Vehicle クラス（および子クラスの Car など）を作ってしまうことです。 車両は本システムにとって「外部存在」であり、システム内でその状態を変化・追跡する必要はありません。 必要なのは「スポットを一致させるための車種ラベル」だけです。よって、クラスではなく単なる Enum で定義するのがベストです。 「名詞をすべてクラス化しない」という判断が、LLD面接官に強くアピールします。

物理状態 vs 関係状態 (Intrinsic vs Relational)

State Isolation

ParkingSpot の状態分離の設計判断です：
・Intrinsic (固有物理状態): スポットIDやサイズはスポット自身が持ちます (id, spotType)。
・Relational (関係状態): 「チケット割り当て中か」はオーケストレーター ParkingLot の `occupiedSpotIds` Set で管理します。 これらを分離し ParkingLot に集約させることで、オブジェクト間の循環参照や整合性エラーを防ぎ、同時並行処理でのスレッド同期を極めてシンプルかつ堅牢に解決可能にします。

お金は必ず Long Cents（整数）で管理

Data Integrity

料金計算で float や double を絶対に用いてはいけません。 コンピューター内部の2進数表現の限界により、0.1 などの小数は厳密に表現できず、計算が重なることで微小な誤差が累積します。 金融系を扱うシステムとしてのプロフェッショナルな姿勢を示すため、金額はすべて最小単位（ドルなら「セント」、円なら「円」）の long / 整数 で取り扱います。

料金計算は Strategy パターンで拡張

Design Patterns

料金計算ロジック（単一時間、車種別、曜日・夜間割引など）を ParkingLot 自体に書くと、要件追加のたびに大本のクラスを変更する必要があり、単一責任原則（SRP）および開閉原則（OCP）に違反します。 これを解決するために PricingStrategy インターフェースを定義し、計算ロジックをクラスとしてカプセル化して依存注入できるようにします。
